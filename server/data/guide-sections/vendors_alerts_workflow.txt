# Vendors, AI document ingestion, renewal alerts, and the calendar

## Vendors and contacts

**UI:** `/vendors` (list), `/vendors/:id` (detail â€” Contracts, Contacts, News, and Communications tabs).
**API:** `server/routes/vendors.js`. **Schema:** `Vendor`, `VendorContact`.

A vendor is one organization. The record carries a co-term complexity rating (`none`, `moderate`, `complex`), notes about how this vendor handles co-terms in practice, support contact info, a budget uplift % (defaults to 10), 1â€“5 ratings on price flexibility, support, strategic value, and overall satisfaction, and an optional list of org-specific aliases.

The vendor name normalizer at `server/lib/vendorNormalizer.js` handles the "Microsoft Corporation / Microsoft / MS" problem. It strips legal-entity suffixes (Inc, LLC, Corp, Ltd, GmbH, etc.) but keeps industry-descriptive words ("Cloud," "Networks," "Technologies"). There's a curated alias map in `server/lib/vendorAliases.js` covering ~90 common cases. The duplicate-vendor warning during create catches near-misses.

Contacts live on the vendor (`VendorContact`), surfaced as the Contacts tab on the vendor detail page. You can add them manually, paste an email signature into the import dialog, or upload a business-card image â€” the AI extracts name, title, email, phone, fax, address, website, notes. See `server/routes/signature.js` and `client/src/pages/VendorDetail.jsx::SignatureImportModal`.

## AI document ingestion

**UI:** `/ingest` (drag-and-drop upload + review queue).
**API:** `server/routes/ingest.js`. Extractor: `server/lib/extractor.js`. AI abstraction: `server/lib/ai.js`.

Pipeline:
1. User drops a file (PDF, Word, plain text, TIFF/JPEG/PNG, or `.lic`). Multer captures it in memory; `server/routes/ingest.js` validates the MIME type against an explicit allowlist *and* magic-byte signature (F014 hardening, May 2026 audit).
2. `extractor.js::extractText` pulls text from the file. PDFs go through `pdf-parse`. Word documents go through `mammoth`. Plain text and `.lic` files are decoded directly. Images skip text extraction and go straight to vision.
3. Extracted text or image is sent to the configured AI provider with a strict JSON schema (`EXTRACTION_SCHEMA` in `extractor.js`). Output is validated by Zod â€” malformed AI responses are rejected, not silently passed through.
4. An `IngestionSession` row is created with raw text, extracted fields, per-field confidence scores, and AI notes explaining anything flagged.
5. User reviews extracted fields in the UI with confidence color-coding. Partial-match handling means if the AI returns `name: null` on a contact, the row still imports rather than failing the whole batch.
6. On approve, the document is linked to the contract record. Source document is preserved.

**Vendor matching during ingest.** The ingest route runs the same `vendorNormalizer` logic the manual-create path uses. Definitive match â†’ contract attached to the existing vendor. High-confidence similar match â†’ user prompted to confirm. Nothing matches â†’ new vendor created.

Per-user-per-day caps in `server/lib/aiQuota.js`.

## Renewal alerts and calendar

**UI:** `/alerts` (full alert center with preferences), notification bell, dashboard "Needs Attention Today" panel.
**API:** `server/routes/alerts.js`. Engine: `server/lib/alertEngine.js`.

Alerts are time-windowed reminders generated by a nightly cron at 07:00 server time. Seven alert types: `review_by`, `cancel_by`, `renewal`, `billing_60`, `billing_30`, `billing_48`, `payment_due`. Each has a per-user preference for which days-before to fire on (default `90,60,30` for review/cancel/renewal).

The engine routes alerts: contracts with an `internalOwnerId` deliver to that owner; contracts without an owner fall back to all account admins. Each recipient gets *one* email per run, not one per alert â€” the goal is a daily summary, not inbox flooding.

Alert preferences live on `AlertPreference` (per-user, per-alert-type). A user can disable email per type, change the days-before list, or both.

**Optional channels.** Slack and Teams webhook digests mirror the email digest. They're per-account configuration in `/settings` and SSRF-protected at the lib layer (`server/lib/slack.js`, `server/lib/teams.js`) â€” webhook URLs are validated against per-provider allowlists.

**Calendar view.** Dashboard's 12-month forward look at three date types simultaneously: `evaluationStartByDate` (review-by), `cancelByDate` (cancel-by), `endDate` (renewal). Color-coded distinctly because confusing them is the most common renewal mistake.
