# Vendors

A Vendor is one organization you buy software from. It's where co-term complexity, support contacts, and vendor news live.

## What you'll see

**Vendor list** at `/vendors`: every vendor, sorted alphabetically. Each row shows total contracts, total annual spend, and a co-term complexity badge.

**Vendor detail** at `/vendors/:id` with four tabs:
- **Contracts** — every contract attached to this vendor, with combined renewal value and a co-term cluster view if any contracts share a `coTermGroup`.
- **Contacts** — VendorContact records: name, title, email, phone, fax, address, website, notes.
- **News** — recent articles classified by the news scanner (security, outage, pricing, EOL, etc.) for this vendor.
- **Communications** — log of calls, meetings, notes, emails against this vendor.

## How vendor normalization works

When a contract is created (manually, via AI ingest, or via CSV import), the vendor name passes through the normalizer. It strips legal-entity suffixes (Inc, LLC, Corp, Ltd, GmbH) but keeps industry-descriptive words ("Cloud", "Networks", "Technologies"). A curated alias map handles ~90 common cases like "Microsoft Corporation" ↔ "Microsoft" ↔ "MSFT".

If a contract's vendor is a definitive match to an existing record, it auto-attaches. If it's a high-confidence near-match, you're prompted to confirm. If nothing matches, a new vendor is created.

## Co-term complexity

Each vendor has a `cotermComplexity` rating: `none`, `moderate`, or `complex`. This describes how the *vendor* handles co-term consolidation in general — Microsoft Enterprise Agreements are `complex`, most SaaS tools are `none`.

This is **separate** from a contract's `coTermGroup` field, which is a per-contract identifier shared across contracts that should be negotiated together (e.g. "Microsoft Q4 2027").

## Common workflows

**"I have three Microsoft records and they should be one."** Pick a canonical name. Open each duplicate, change `vendorId` on each of its contracts (bulk reassign via the contracts list, filtered by the duplicate vendor). Then add the duplicates as aliases on the canonical record via Settings → Vendor Aliases. Archive the now-empty duplicate vendors.

**"I need a per-vendor renewal pipeline."** Go to `/vendors/:id`, Contracts tab. You'll see combined renewal value for the next 90 / 180 / 365 days plus a cluster view if any contracts share a co-term group.

**"How do I add a new contact from an email signature?"** On the vendor detail Contacts tab, click "Import from signature". Paste the email signature text OR upload a business-card image — AI extracts name, title, email, phone, fax, address, website, and notes for you to review before saving.

**"Where do I record that we had a call with the vendor about price?"** Vendor detail → Communications tab → "Log entry". Pick the type (call, meeting, note, email), set the date, write the note. The log survives staff turnover — when the renewals manager who knew the vendor leaves, this is what their replacement reads.

## When something looks wrong

**A new contract created a duplicate vendor instead of attaching to an existing one.** The normalizer's confidence threshold didn't match. Add the variant name as an alias on the canonical vendor, then move the new contract to it.

**Vendor news isn't surfacing for a vendor I care about.** The news scanner matches articles against `Vendor.name` + the curated alias list. If your vendor name is unusual or has variants the alias map doesn't know, news might miss. Add aliases as needed. Status-page items (AWS, Azure, etc.) use a forced-attribution path — those work even without name matches.

**A vendor I deleted still shows up in old contracts.** Vendors aren't hard-deleted; they're archived. The old contracts retain their `vendorId` link to the archived vendor record, which still exists in the database for historical accuracy.
