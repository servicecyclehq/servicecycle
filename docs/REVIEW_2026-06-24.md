## Critical (fix immediately)

### C1 — `parts`/`spare_inventory` tables are never created: the migration is empty *(FIXED)*

**What:** `server/prisma/migrations/20260623120000_parts_spare_inventory/` exists as a directory but
contained **no `migration.sql`** — it was the only empty migration dir in the tree (every other
migration, including `20260623000000_arc_flash_incident_register`, has its SQL). The `Part` and
`SpareInventory` models were added to `schema.prisma`, but no DDL was ever generated/committed.

**Why it matters:** `prisma migrate deploy` records the (empty) migration as applied and runs **no
SQL**, so the `parts` and `spare_inventory` tables are never created. Every `/api/parts` write then
fails — confirmed empirically: `parts.test.ts` returns 500 with
*"The table `public.parts` does not exist in the current database"* (18/20 tests fail; only the
no-DB 400/403 cases pass). The entire Chunk-4 feature is dead-on-arrival on any environment migrated
from the committed history. (By contrast `v1Telemetry.test.ts` — whose tables have a real migration —
passes 10/10, isolating this as a migration-only defect.)

**Fix applied:** regenerated the exact DDL from the Prisma models using `prisma migrate diff` (verified
identical to Prisma's own output) and wrote it to
[`20260623120000_parts_spare_inventory/migration.sql`](../server/prisma/migrations/20260623120000_parts_spare_inventory/migration.sql):
both `CREATE TABLE`s, all 7 indexes, and all 5 foreign keys with the correct `onDelete` semantics
(`parts.account` Cascade; `spare_inventory` → part Cascade, → asset SetNull, → site SetNull).
`prisma validate` passes.

> **Operational caveat (read before deploy):** I did **not** run the migration against any database
> (out of scope / shared infra). On a normal environment that has **not** yet recorded this migration,
> `prisma migrate deploy` will now create the tables correctly. If an environment already recorded the
> *empty* `20260623120000_parts_spare_inventory` in `_prisma_migrations`, adding SQL changes the
> checksum and `migrate deploy` will report *"migration was modified after applied."* Resolve that env
> with `prisma migrate resolve --rolled-back 20260623120000_parts_spare_inventory` and re-deploy (or, if
> the row exists but no tables were created, apply this SQL once and `migrate resolve --applied`). Verify
> the demo/prod box state before deploying.

## High (fix before next demo)

### H1 — `consultant` and cross-account roles can create/send quote requests *(FIXED)*

**What:** `POST /api/quote-requests` (create) and `POST /api/quote-requests/:id/send` (promote a
draft) had **no role guard** at all — only `authenticateToken` from the mount in `index.ts`. The
only quote-request route that was gated is `PATCH /:id/status` (`requireManager`).

**Why it matters:** the `consultant` role is an **external** maintenance-vendor login that is
documented and advertised as read-only: `middleware/roles.ts` states *"All write paths: BLOCKED.
The same 403 plain viewers get,"* and the SPA shows a *"Vendor Access — Changes are logged"* banner
([Sidebar.jsx:217](../client/src/components/Sidebar.jsx)). `requireManager` (used everywhere else to
enforce that promise) excludes `consultant`, so the missing guard let a consultant **write**.
A consultant `POST /` does not merely create a row — for a non-draft it fires
`emitQuoteCreatedEvent` ([quoteRequests.ts:51](../server/routes/quoteRequests.ts)), an **outbound
contractor partner event / email** enriched with the full asset dossier **and the account's
portfolio-rank talking points** (rank, percentile, maturity level). An external read-only party
could therefore trigger outbound comms and leak portfolio standing. The cross-account read-only
roles (`oem_admin`, `group_admin`, `super_admin`) were likewise unblocked.

**Note on the brief's premise:** the task expected `field_tech` to reach these routes. It cannot —
`field_tech` is denied at the universal auth chokepoint (`authenticateToken` → `isFieldTechAllowed`),
and `/api/quote-requests` is **not** on the `FIELD_TECH_ALLOWED_PREFIXES` allowlist
([fieldRoleScope.ts:30](../server/lib/fieldRoleScope.ts), [auth.ts:120](../server/middleware/auth.ts)).
So the real exposure is `consultant` + the cross-account roles, not `field_tech`.

**Fix applied:**
- Added a `requireRole(roles)` factory to [`middleware/roles.ts`](../server/middleware/roles.ts)
  (audit-logs the denial like the other gates, returns 403).
- In [`routes/quoteRequests.ts`](../server/routes/quoteRequests.ts) added
  `requireQuoteWriter = requireRole(['admin','manager','viewer'])` and applied it to `POST /` and
  `POST /:id/send`.

This **preserves** the intended internal flow — a `viewer` (internal facility staff) can still raise
a quote from an asset, matching the un-gated **Request Quote** button in
[`QuoteRequestButton.jsx`](../client/src/components/QuoteRequestButton.jsx) — while giving the
external `consultant` and the cross-account roles the same 403 every other write path gives them.
`PATCH /:id/status` was left at `requireManager` (status transitions are a rep/manager action).

> Regression test to add (could not run here — see Task 5): a `consultant` `POST /api/quote-requests`
> must return 403; a `viewer` must still return 201. The existing
> `quoteDraftDisasterGate.test.ts` already exercises the manager create/send path and is unaffected.

## Medium (fix this sprint)

### M1 — Telemetry auto-quote inside the `$transaction` is **not** actually non-fatal

**Where:** [telemetryMonitoring.ts:103-137](../server/lib/telemetryMonitoring.ts), invoked from the
per-reading `prisma.$transaction(...)` at [v1/telemetry.ts:146-149](../server/routes/v1/telemetry.ts).

**What / trace:** the auto-quote block wraps `db.quoteRequest.create` in `try/catch` and the comment
claims *"Non-fatal — failure logs but does not block the condition state update."* That is **false**
for any real database error. Inside a Postgres interactive transaction, a failed statement aborts the
whole transaction at the DB level — every later command (and the final `COMMIT`) errors with
*"current transaction is aborted."* Catching the JS error does not un-poison the PG transaction.
Concretely:

1. `quoteRequest.create` throws (e.g. FK violation if the chosen requester is hard-deleted between the
   in-tx `user.findFirst` and the `create`, or any constraint/connection error).
2. The `catch` logs and returns; `applyMonitoringState`/`ingestReading` return normally.
3. The tx callback resolves, Prisma issues `COMMIT` → **fails** (tx poisoned).
4. `await prisma.$transaction(...)` **rejects**. There is no per-reading `try/catch` in the loop, so
   it propagates to the outer `catch` ([v1/telemetry.ts:159](../server/routes/v1/telemetry.ts)) → the
   reading is rolled back (the reading itself is lost), the batch loop aborts, the whole request 500s,
   and the idempotency record is never stored ([line 157](../server/routes/v1/telemetry.ts)) → a retry
   re-runs and, if deterministic, becomes a poison-pill.

**Probability:** low — the inputs are validated, the requester is freshly looked up in-tx and
null-guarded, the enum casts (`suspected_failing`/`within_1_week`) are valid `QuoteDriver`/
`QuoteTimeline` values, and there are no unique constraints to violate. So this rarely fires — but
when it does the blast radius is the entire batch, and the "non-fatal" guarantee is simply wrong.

**Fix (describe — non-trivial, spans files):** move the quote creation **out** of the transaction.
Have `applyMonitoringState` return a `escalatedToMonitoring: true` signal on the `false→true`
transition instead of creating the quote; thread it through `ingestReading`'s return; then in the
route, **after** `await prisma.$transaction(...)` resolves, call a `createTelemetryCritQuote(prisma,
…)` helper using the non-tx client. A post-commit failure then truly cannot roll back the reading.
This also resolves M3.

### M2 — Auto-quote idempotency over-suppresses (the brief's concern is inverted)

**Where:** [telemetryMonitoring.ts:105-108](../server/lib/telemetryMonitoring.ts).

**What:** the dedup query is `{ triggerType: 'TELEMETRY_CRIT', status: { not: 'declined' } }`. The
brief worried a new quote *would* be created after a prior was `accepted`. The code does the
**opposite**: `accepted` is *not* `declined`, so an accepted (i.e. **resolved**) prior quote
**matches** and **suppresses** all future auto-quotes for that asset — forever, unless someone
declines the old one. A genuinely new CRIT episode months later (after the previous issue was
serviced/accepted) silently gets **no** quote. (Re-escalation while a quote is still
`requested`/`quoted` is correctly deduped — verified by the existing
`v1Telemetry.test.ts` "manual acknowledge … still only 1 QuoteRequest" case.)

**Fix (describe):** scope the idempotency window to in-flight statuses so resolution doesn't block a
new episode:
```ts
where: { accountId, assetId: asset.id, triggerType: 'TELEMETRY_CRIT',
         status: { in: ['draft', 'requested', 'quoted'] } },
```
Optionally add a recency window (e.g. last N days) if you want to throttle rapid re-CRITs even while
in-flight. Confirm intended product behavior with the team first.

### M3 — Two extra in-transaction queries lengthen each reading's tx

**Where:** the `user.findFirst` + `quoteRequest.findFirst` (+ `create`) added inside the per-reading
interactive transaction ([telemetryMonitoring.ts:105-131](../server/lib/telemetryMonitoring.ts)).

**What:** interactive transactions carry a default 5 s timeout. These queries only run on the rare
`false→true` transition, so per-reading timeout risk is small, but they extend a transaction that
already does ~8-10 statements. The correct home for a side-effect that "must not fail the write" is
**outside** the tx — the M1 fix moves them there and removes this concern too.

## Low / Suggestions

- **L1 — `qtyOnHand` silently clamps bad input to 0.** In `POST /:id/inventory`,
  `posInt(req.body?.qtyOnHand) ?? 0` turns a negative or `> INT_MAX` value into `0` rather than a 400
  ([parts.ts:239](../server/routes/parts.ts)). Safe (no overflow/500) but masks client bugs; consider
  rejecting clearly-invalid quantities.
- **L2 — Client inbox is stricter than the server read gate.** The `/quote-requests` page is
  `RequireRole ['admin','manager']` ([App.jsx:489](../client/src/App.jsx)) but server `GET
  /api/quote-requests` has no role gate, so a `viewer`/`consultant` can read their **own account's**
  quotes via the API while being hidden from the UI. Reads are account-scoped (no tenant leak) — this
  is a cosmetic client-vs-server mismatch, not a security hole. Decide intent: relax the client gate if
  viewers should see the inbox, otherwise leave as-is.
- **L3 — Stale schema comment.** `QuoteRequest.triggerType` doc lists
  `MODERNIZATION_EOL | ARC_FLASH_STUDY | QEMW_TRAINING` but not `TELEMETRY_CRIT`, which
  `telemetryMonitoring.ts` now writes ([schema.prisma:2296](../server/prisma/schema.prisma)). Update the
  comment.
- **L4 — `risk-score` incident `take: 500`.** Sufficient: the incident penalty saturates at
  `INCIDENT_MAX_PENALTY` (15) with as few as 3 injury or 8 non-injury incidents, so under-counting
  beyond 500 **cannot** change the score. Only the displayed `openWithInjury/openNoInjury` metrics would
  under-count for a pathological >500-open-incident account. If exactness of the displayed metric
  matters, use `count()` with an `injury` filter instead of slicing a `findMany`.
